# OKF v0.1 — Spec Digest (authoritative condensation)

> Condensed from the official Open Knowledge Format v0.1 specification
> (GoogleCloudPlatform/knowledge-catalog · `okf/SPEC.md`). When this digest and the
> upstream SPEC disagree, **the upstream SPEC wins**. This file exists for progressive
> disclosure: load it instead of the full spec, and only when you need the rules.
>
> **Digest § ↔ upstream §.** This digest uses its own numbered headings; each names the
> upstream section it condenses. Tooling and [conformance.md](conformance.md) always cite
> the **upstream** numbers: `§3` document · `§4` fields · `§5` links · `§6` index ·
> `§7` log · `§8` citations · `§9` conformance · `§10` versioning · `§11` root `okf_version`.

## 1. What OKF is (upstream §1–§2)

OKF is a **format, not a platform**. A *bundle* is a directory tree of UTF-8 Markdown
files with YAML frontmatter, plus a few agreed conventions, so that knowledge written by
one producer can be consumed by any agent or human without translation. It formalizes the
"LLM-wiki" pattern. It is minimally opinionated: it standardizes the *interoperability
surface*, not the content model.

## 2. The one hard rule (upstream §9)

> Every non-reserved `.md` file (a **concept document**) MUST have a parseable YAML
> frontmatter block whose `type` field is a non-empty string.

Everything else below is recommended or conventional. `type` is the **only** required field.

## 3. Concept documents (upstream §3–§4)

A concept document is UTF-8 Markdown with two parts:

1. **YAML frontmatter** — delimited by a line `---` at the very start of the file and a
   closing line `---`.
2. **Markdown body** — free-form content after the closing `---`.

### Frontmatter fields

| Field         | Status      | Meaning |
|---------------|-------------|---------|
| `type`        | **REQUIRED** | Short string naming the concept kind (e.g. `BigQuery Table`, `Metric`, `Playbook`). Not centrally registered; choose descriptive values. Consumers MUST tolerate unknown types. |
| `title`       | recommended | Human-readable display name. If omitted, derive from filename. |
| `description` | recommended | Single sentence summarizing the concept. |
| `resource`    | recommended | URI uniquely identifying the underlying asset. Absent for abstract concepts. |
| `tags`        | recommended | YAML list of short strings for cross-cutting categorization. |
| `timestamp`   | recommended | ISO 8601 datetime of the last meaningful change (e.g. `2026-05-28T14:30:00Z`). |

Producers MAY add any additional keys. Consumers MUST preserve unknown keys when
round-tripping and MUST NOT reject documents that carry them.

### Concept ID

A concept's identity is its **file path within the bundle with the `.md` suffix removed**.
`tables/orders.md` → concept ID `tables/orders`.

### Conventional body headings (soft guidance, none required)

| Heading       | Purpose |
|---------------|---------|
| `# Schema`    | Structured description of an asset's columns/fields. |
| `# Examples`  | Concrete usage examples, often fenced code blocks. |
| `# Citations` | Numbered external sources backing claims (`[1] [Title](url)`). |

## 4. Reserved filenames (upstream §6 index, §7 log)

Only two filenames are reserved. They are **not** concept documents.

### `index.md`
- A directory listing supporting progressive disclosure. May appear in any directory,
  including the bundle root.
- **No frontmatter**, with one exception: the **bundle-root** `index.md` MAY carry a single
  frontmatter key, `okf_version` (e.g. `okf_version: "0.1"`), to declare the target spec
  version. This is the only place frontmatter is permitted in an `index.md`.
- Body groups concepts under section headings, each entry a Markdown link plus a short
  description, ideally sourced from the linked concept's `description`:
  ```markdown
  # Tables
  * [Orders](orders.md) - One row per completed customer order.
  * [Customers](customers.md) - One row per customer.
  ```
- Index files MAY be auto-generated by producers or synthesized on the fly by consumers.

### `log.md`
- Optional change history at any hierarchy level. Flat list of date-grouped entries,
  **newest first**, with ISO 8601 `YYYY-MM-DD` date headings:
  ```markdown
  # Update Log

  ## 2026-05-22
  * **Update**: Refined the join description for orders↔customers.
  * **Creation**: Added the customers table.
  ```
- The leading bold word (Update, Creation, Deprecation, …) is convention, not a requirement.

## 5. Cross-links (upstream §5, citations §8)

Concepts link to each other with **normal Markdown links**. A link from concept A to
concept B asserts an (untyped) directed relationship; the *kind* of relationship is conveyed
by the surrounding prose, not the link.

- **Absolute / bundle-relative** links begin with `/` and resolve from the bundle root:
  `[customers](/tables/customers.md)`. **Recommended** — stable when files move between
  subdirectories.
- **Relative** links are ordinary Markdown relative paths: `[other](./other.md)`.
- Consumers **MUST tolerate broken links** — a missing target is not malformed; it may be
  knowledge not yet written.

### Citations
Listed under a `# Citations` heading at the document bottom, numbered. Targets may be
absolute URLs, bundle-relative paths, or paths into a `references/` subdirectory that
mirrors external material.

## 6. Conformance (upstream §9)

A bundle is **conformant** with OKF v0.1 iff:

1. Every non-reserved `.md` file contains a parseable YAML frontmatter block; **and**
2. Every such frontmatter block contains a non-empty `type` field; **and**
3. Reserved files (`index.md`, `log.md`) follow their structure (§4) when present.

### The permissive consumption model (do NOT treat these as errors)
Consumers **MUST NOT** reject a bundle for any of:
- missing optional/recommended frontmatter fields;
- unknown `type` values;
- unknown additional frontmatter keys;
- broken cross-links;
- missing `index.md` files.

All other constraints in the spec are **SHOULD**-level soft guidance.

## 7. Versioning (upstream §10–§11)

- Version string is `<major>.<minor>`; current is `0.1`.
- **Minor** bump = backward-compatible additions (new optional fields, conventional headings).
- **Major** bump = breaking changes (renamed required fields, changed reserved filenames).
- A bundle MAY declare its target version via `okf_version` in the bundle-root `index.md`
  frontmatter. Consumers should make a best-effort attempt rather than refuse undeclared bundles.

## 8. Distribution (upstream §2)

A bundle is just files. Ship it as a git repository (recommended), a tar/zip archive, or a
subdirectory inside a larger repository. No accounts, SDKs, or services are required to
produce or consume one.
